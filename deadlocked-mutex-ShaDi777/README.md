# Взаимное исключение и обнаружение дедлоков

В рамках данной лабораторной работы необходимо реализовать:

1. Распреденный мьютекс по алгоритму Maekawa
2. Распределённый снимок состояния системы по алгоритму Chandy–Lamport
3. Обнаружение дедлока на основе глобального состояния (построение wait-for графа и поиск циклов)

Работа выполняется на Go с использованием фреймворка [Hive](https://github.com/nikitakosatka/hive).

---

## Контекст

В распределённой системе несколько узлов хотят входить в критическую секцию, при этом:

- нет общего центрального координатора
- узлы обмениваются сообщениями по сети
- каждый узел знает только ограниченное подмножество других узлов (кворум)

Задача — построить поверх такой системы:

- **распределённый мьютекс**, который не допускает одновременного входа двух процессов в критическую секцию
- **распределённый снимок состояния**, который фиксирует согласованный глобальный срез системы
- **детектор дедлоков**, который по снимку определяет существование цикла в графе ожиданий

---

## Что нужно реализовать

Вам необходимо реализовать:

```go
func NewMaekawaMutexNode(id string, allNodeIDs []string) MutexNode
```

а также методы интерфейса:

```go
type MutexNode interface {
    hive.Node

    // Lock blocks until the node enters the critical section.
    Lock() error

    // Unlock releases the critical section.
    Unlock() error

    // InCriticalSection reports whether this node currently holds the mutex.
    InCriticalSection() bool
}
```

и функции/методы, связанные со снимком и взаимоблокировками

```go
// Распределённый снимок по Чанди–Лэмпорту
func Snapshot(nodes []MutexNode) (GlobalState, error)

// Анализ глобального состояния
func (s GlobalState) HasDeadlock() bool
func (s GlobalState) IsConsistentCut() bool
```

Типы `Event`, `LocalState`, `GlobalState` и структура `MaekawaMutexNode` уже определены и должны быть использованы в вашей реализации, но могут быть модифицированы.

## Требования

### 1. Mutex

Необходимо реализовать протокол распределённой взаимной исключительности на основе алгоритма Maekawa:

- сформировать **систему кворумов** на основе `allNodeIDs`, такую что:
    - каждый узел имеет свой кворум (подмножество узлов)
    - любые два кворума **пересекаются** хотя бы в одном узле
- при вызове `Lock()`:
    - узел рассылает запросы (запрос на захват мьютекса) узлам из своего кворума
    - получает разрешения от кворума
    - входит в критическую секцию только после получения достаточного количества разрешений
- при вызове `Unlock()`:
    - узел освобождает свой “билет” в кворуме
    - даёт возможность другим ожидающим узлам войти в критическую секцию

Необходимо обеспечить:

- **взаимную исключительность**: не более одного узла в критической секции в каждый момент времени
- **корректность протокола**: узлы не могут обходить протокол и входить в критическую секцию без голосов кворума

### 2. Согласованный срез (Chandy–Lamport)

Необходимо реализовать алгоритм Chandy–Lamport для построения **согласованного глобального среза**:

- `Snapshot(nodes []MutexNode) (GlobalState, error)` должен:
    - инициировать протокол снимка на одном из узлов
    - разослать **маркерные сообщения** всем остальным узлам
    - на каждом узле один раз записать локальное состояние (первые `k` событий `Events` для `LocalState`)
    - корректно обрабатывать сообщения, приходящие:
        - до записи локального состояния
        - после записи, но до получения маркера по соответствующему каналу (состояние канала)
    - завершить протокол, когда все узлы завершили локальную часть снимка
    - вернуть `GlobalState`, содержащий локальные состояния всех узлов и их идентификаторы

Свойство согласованности:

- метод `IsConsistentCut()` на `GlobalState` должен возвращать `true` для корректно собранных снимков:
    - не должно быть события “приёма сообщения” в срезе, если соответствующее событие “отправки сообщения” не попало в срез
- для искусственно некорректных срезов (в тестах) метод должен выявлять несогласованность и возвращать `false`

### 3. Обнаружение взаимоблокировок

На основе `GlobalState` необходимо реализовать обнаружение взаимоблокировок:

- метод `HasDeadlock()` должен:
    - построить **wait-for граф** (граф ожиданий):
        - вершины — узлы системы
        - ребро `A -> B` означает, что узел A ждёт ресурса (разрешения/голоса) у узла/кворума, который в данный момент “удерживается” в пользу B
    - использовать информацию о:
        - отправленных и полученных сообщениях протокола Маэкавы (запросы, разрешения, освобождения)
        - текущем состоянии голосов кворумов (кому сейчас отдан голос каждого участника)
- наличие цикла в wait-for графе трактуется как **взаимоблокировка**:
    - если есть цикл ожиданий, `HasDeadlock()` должен вернуть `true`
    - иначе — `false`
